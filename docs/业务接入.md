# 业务接入

本文档描述面向业务系统的 `/api` 前缀接口。

这组接口用于业务侧服务接入 IM，例如业务注册用户、更新昵称头像、查询用户信息、签发 IM 连接 token、代用户创建会话、查询会话数据、管理成员和会话控制、发送消息。

## 1. 鉴权方式

所有 `/api/*` 接口都需要在请求头中传递：

```http
X-App-Code: app_xxxxxxxx
X-App-Key: your_app_key
Content-Type: application/json
```

说明：

- `X-App-Code`：应用编码
- `X-App-Key`：应用密钥
- 这组接口适合业务后端服务调用，不建议直接暴露给公开客户端

## 2. 创建或更新用户

- 方法：`POST`
- 路径：`/api/users`

请求体：

```json
{
  "external_user_id": "u_1001",
  "nickname": "张三",
  "avatar_url": "https://example.com/avatar.png"
}
```

成功响应：

```json
{
  "code": 200,
  "data": {
    "id": 1,
    "app_id": 1,
    "external_user_id": "u_1001",
    "nickname": "张三",
    "avatar_url": "https://example.com/avatar.png",
    "status": "active",
    "created_at": "2026-07-09T11:00:00+08:00",
    "updated_at": "2026-07-09T11:00:00+08:00"
  },
  "message": "",
  "meta": {
    "timestamp": "2026-07-09T11:00:00+08:00",
    "response_time": 2
  }
}
```

说明：

- 当用户不存在时会创建
- 当用户已存在时会更新昵称和头像
- 已有用户状态会被保留，不会因为更新资料而自动解除禁用

成功响应：

```json
    "app_id": 1,
    "external_user_id": "u_1001",
    "nickname": "张三",

    "status": "active",

    "updated_at": "2026-07-09T11:00:00+08:00"
  },
  "message": "",
  "meta": {
    "timestamp": "2026-07-09T11:00:00+08:00",
    "response_time": 1
  }
}
```

## 4. 更新用户资料

- 方法：`PUT`
- 路径：`/api/users/:external_user_id`

请求体：

```json
{
  "nickname": "李四",
  "avatar_url": "https://example.com/avatar-new.png"
}
```

说明：

- 字段按需传递
- 未传字段会保留原值
- 该接口只更新用户资料，不修改用户状态

## 5. 签发 IM Token

- 方法：`POST`
- 路径：`/api/users/:external_user_id/token`

成功响应：

```json
{
  "code": 200,
  "data": {
    "app_code": "app_xxxxxxxx",
    "user_id": "u_1001",
    "token": "jwt_token",
    "expires_at": "2026-07-10T11:00:00+08:00"
  },
  "message": "",
  "meta": {
    "timestamp": "2026-07-09T11:00:00+08:00",
    "response_time": 1
  }
}
```

说明：

- 只有用户存在且状态为 `active` 时才能签发 token
- 返回的 token 可直接用于：
  - HTTP IM 接口：`Authorization: Bearer <token>`
  - WebSocket：`/im/connect?token=<token>`

## 6. 业务侧创建会话

- 方法：`POST`
- 路径：`/api/conversations`

请求体：

```json
{
  "type": "group",
  "subject": "项目群",
  "owner_user_id": "u_1001",
  "member_user_ids": ["u_1002", "u_1003"]
}
```

说明：

- `owner_user_id` 为业务侧指定的会话创建者
- 会复用 IM 会话创建规则，例如 `single` 只能带 1 个成员，`chatroom` / `live_room` 不允许初始化成员

### 6.1 创建职责建议

会话创建推荐走：

```text
zcyp-web -> zcyp-backend -> zcyp-im
```

不建议由前端直接调用 `/api/conversations` 或 `/im/conversations`。原因是创建会话通常不只是 IM 动作，还会涉及业务对象绑定、幂等校验、权限判断和会话映射落库。

### 6.2 `conversation_no` 落库建议

业务后端在成功创建会话后，建议保存业务对象与 `conversation_no` 的映射关系，例如：

```text
business_type + business_id + scene -> conversation_no
```

常见示例：

- `order + 10001 + service -> conv_xxx`
- `project + 88 + group -> conv_xxx`
- `live_room + 9527 + room -> conv_xxx`

后续查询会话、拉历史消息、成员管理、发消息前，业务后端都应优先根据自己的业务主键查出 `conversation_no`，再调用 IM 接口。

### 6.3 单聊查重与幂等创建

`zcyp-im` 当前不会根据业务主键、用户对或外部幂等键自动去重。对于 `single` 会话，推荐由业务后端负责“先查后建”。

推荐做法：

1. 业务后端对单聊双方生成稳定的 `peer_key`
2. 先在业务库查 `peer_key` 是否已经映射到某个 `conversation_no`
3. 若已存在，直接复用原会话
4. 若不存在，再调用 `POST /api/conversations`
5. 创建成功后，把 `peer_key -> conversation_no` 落库

`peer_key` 生成建议：

```text
sort(user_a, user_b) -> user_a:user_b
```

例如：

```text
u_1001:u_1002
```

这样 A 发起和 B 发起会得到同一个单聊键，避免重复创建两个单聊会话。

### 6.4 群聊 / 业务群幂等建议

对于 `group`、`chatroom`、`live_room`，推荐由业务后端基于业务对象自己保证幂等，而不是依赖 IM 服务判断“这是不是同一个群”。

常见做法：

- 一个订单只允许一个客服群：`order_id + scene`
- 一个项目只允许一个讨论组：`project_id + scene`
- 一个直播房间只允许一个直播会话：`room_id + scene`

如果业务允许一个对象创建多个不同会话，应再增加业务维度，例如 `scene`、`channel`、`topic`。

## 7. 业务侧查询历史消息

- 方法：`GET`
- 路径：`/api/conversations/:conversation_no/messages`

查询参数：

- `user_id`：必填，表示以哪个用户身份查询
- `limit`：可选，默认 `50`，范围 `1-200`

说明：

- 只有会话成员可查询
- 会复用 IM 侧的成员权限校验

## 8. 业务侧查询成员列表

- 方法：`GET`
- 路径：`/api/conversations/:conversation_no/members`

查询参数：

- `user_id`：必填，表示以哪个用户身份查询

说明：

- 只有会话成员可查询
- 会复用 IM 侧的成员权限校验

## 9. 业务侧成员管理与会话控制

以下接口适合业务后端代用户执行成员管理或会话控制动作，都会复用 IM 侧的权限校验规则。请求体里统一显式传 `operator_user_id`。

### 9.1 添加成员

- 方法：`POST`
- 路径：`/api/conversations/:conversation_no/members`

请求体：

```json
{
  "operator_user_id": "u_1001",
  "member_user_ids": ["u_1002", "u_1003"]
}
```

### 9.2 移除成员

- 方法：`DELETE`
- 路径：`/api/conversations/:conversation_no/members/:member_user_id`

请求体：

```json
{
  "operator_user_id": "u_1001"
}
```

### 9.3 全员禁言开关

- 方法：`POST`
- 路径：`/api/conversations/:conversation_no/all-muted`

请求体：

```json
{
  "operator_user_id": "u_1001",
  "enabled": true
}
```

### 9.4 内容审核开关

- 方法：`POST`
- 路径：`/api/conversations/:conversation_no/review`

请求体：

```json
{
  "operator_user_id": "u_1001",
  "enabled": true
}
```

### 9.5 封禁 / 解封成员

- `POST /api/conversations/:conversation_no/members/:member_user_id/ban`
- `POST /api/conversations/:conversation_no/members/:member_user_id/unban`

请求体：

```json
{
  "operator_user_id": "u_1001"
}
```

### 9.6 成员角色与麦克风

- `POST /api/conversations/:conversation_no/members/:member_user_id/role`
- `POST /api/conversations/:conversation_no/members/:member_user_id/mic`

请求体示例：

```json
{
  "operator_user_id": "u_1001",
  "role": "speaker"
}
```

```json
{
  "operator_user_id": "u_1001",
  "mic_status": "on"
}
```

### 9.7 禁言 / 解除禁言

- `POST /api/conversations/:conversation_no/members/:member_user_id/mute`
- `POST /api/conversations/:conversation_no/members/:member_user_id/unmute`

请求体示例：

```json
{
  "operator_user_id": "u_1001",
  "minutes": 30
}
```

```json
{
  "operator_user_id": "u_1001"
}
```

## 10. 业务侧发送消息

- 方法：`POST`
- 路径：`/api/conversations/:conversation_no/messages`

请求体：

```json
{
  "sender_user_id": "u_1001",
  "message_type": "text",
  "client_msg_id": "client_001",
  "content": "hello"
}
```

成功响应：

```json
{
  "code": 201,
  "data": {
    "id": 1,
    "message_no": "msg_xxxxxxxx",
    "app_id": 1,
    "conversation_id": 1,
    "sender_user_id": "u_1001",
    "message_type": "text",
    "client_msg_id": "client_001",
    "content": {
      "type": "text",
      "text": "hello"
    },
    "created_at": "2026-07-10T11:05:00+08:00"
  },
  "message": "",
  "meta": {
    "timestamp": "2026-07-10T11:05:00+08:00",
    "response_time": 1
  }
}
```

说明：

- 该接口适合业务后端代用户发消息，不需要用户 Bearer Token
- 仍会复用 IM 层的成员校验、禁言、全员禁言、直播间发言权限和内容审核规则
- 发送成功后会同步广播给已订阅该会话的 WebSocket 客户端

## 11. 推荐接入流程

推荐的业务接入链路如下：

### 11.1 建立单聊或群聊

1. `zcyp-web` 请求 `zcyp-backend`
2. `zcyp-backend` 根据业务对象或 `peer_key` 查询本地映射表
3. 若已存在 `conversation_no`，直接返回
4. 若不存在，调用 `POST /api/conversations`
5. 把返回的 `conversation_no` 保存到业务库
6. 返回给前端

### 11.2 获取 IM token 并建立长连接

1. `zcyp-web` 请求 `zcyp-backend`
2. `zcyp-backend` 调用 `POST /api/users/:external_user_id/token`
3. `zcyp-backend` 把 token 返回给前端
4. `zcyp-web` 直接连接 IM WebSocket 网关：

```text
wss://<im-host>/im/connect?token=<token>
```

### 11.3 后续操作分层

- 会话创建、映射查询、业务权限判断：走 `zcyp-backend`
- WebSocket 建连、订阅、实时收消息：前端直连 `zcyp-im`
- 业务侧代用户成员管理、发消息、查历史：`zcyp-backend -> zcyp-im /api/*`

## 12. 常见错误

应用鉴权失败：

```json
{
  "code": 401,
  "data": null,
  "message": "invalid app credentials",
  "meta": {
    "timestamp": "2026-07-09T11:00:00+08:00",
    "response_time": 0
  }
}
```

用户不存在：

```json
{
  "code": 404,
  "data": null,
  "message": "user not found",
  "meta": {
    "timestamp": "2026-07-09T11:00:00+08:00",
    "response_time": 0
  }
}
```

用户已禁用：

```json
{
  "code": 403,
  "data": null,
  "message": "user disabled",
  "meta": {
    "timestamp": "2026-07-09T11:00:00+08:00",
    "response_time": 0
  }
}
```
